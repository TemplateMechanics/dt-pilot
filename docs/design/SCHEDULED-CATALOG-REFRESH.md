## Problem

`config/catalog/catalog.settings.json` is hand-maintained. Today there are 8 entries covering common Dynatrace settings 2.0 schemas. The catalog tells the agent what scaffolds exist and informs the `commonParameters` pre-declared in each scaffold.

Three failure modes follow from manual maintenance:

1. **Bit-rot.** Schemas evolve. `builtin:alerting.profile` could gain a new severity level next quarter; the catalog wouldn't know.
2. **Coverage drift.** New, useful schemas ship (notebooks, automation workflows, security findings) and no one notices because no one runs `monaco generate schema --list` periodically.
3. **Curation cost as a tax on growth.** Adding the 9th entry costs the same focused PR as the 1st did. Every team that wants its specific schema covered has to author a `chore/catalog-add-<schema>` PR, run the generator, commit the regen, get review. That cost is what kills catalogs in practice — they ossify at whatever the original author cared about.

The fix is to make the catalog **mostly autogenerated** from a small input list, refreshed on a schedule, with humans only reviewing diffs and approving the auto-PR.

## Goals

- **Catalog freshness without scheduled human effort.** A weekly cron updates the catalog `summary` text and the (informational) `liveFields` array from `monaco generate schema` output. `commonParameters` stays hand-curated — see Non-goals for why. Drift between live Dynatrace and the committed catalog never exceeds one week.
- **Coverage growth is one-line, not one-PR.** Adding a schema to the inputs list (`config/catalog/schemas.txt` — one schema ID per line) is enough; the cron picks it up on the next run.
- **Humans review machine-generated changes, not author them.** The cron opens a PR; a human (or future agent) reviews the diffstat and merges. No silent commits to `main`.
- **The refresh job never deploys anything.** It reads schemas (via `monaco generate schema`); it does not call `deploy`, `delete`, or `download`. Lowest-privilege Dynatrace credentials.

## Non-goals

- Auto-generating the catalog from *every* Dynatrace schema. Two reasons: (1) there are too many and most are operationally irrelevant; (2) a hand-curated inputs list keeps the catalog opinionated about what's worth supporting.
- Auto-curating `commonParameters` purely from the schema. The schema knows every field; "common" is a judgment call. The cron updates `summary` and a `liveFields` list (informational); a maintainer still decides which fields are `commonParameters` worth pre-declaring.
- Auto-merging the catalog refresh PR. Merge is a deliberate review, every time, even when the diff is small.
- Per-backend autogeneration. This proposal covers Monaco only; equivalent jobs for Terraform / Crossplane / Pulumi catalogs come with their own backend PRs.

## Proposed design

### Inputs file — `config/catalog/schemas.txt`

```
# One Dynatrace settings 2.0 schema ID per line. Lines starting with #
# are comments. Blank lines ignored. The order here determines the order
# entries appear in the regenerated catalog.

builtin:management-zones
builtin:tags.auto-tagging

builtin:alerting.profile
builtin:slo
builtin:problem.notifications
builtin:problem.notifications/email
builtin:anomaly-detection.metric-events

builtin:dashboards
```

This file is the single source of truth for "what does dt-pilot ship a scaffold for?". Adding a row = adding scaffold coverage. Removing a row = removing it.

### Refresh script — `scripts/monaco/Sync-CatalogFromSchemas.ps1`

Runs locally and in CI. Behavior:

1. Read `config/catalog/schemas.txt`.
2. For each schema ID, call `monaco generate schema --schema <id> --output -` (pipe to stdout) via the existing `Invoke-MonacoGenerate.ps1` infrastructure, parsing the resulting JSON Schema.
3. Build a new catalog entry per schema, **preserving the existing `family` and `commonParameters` values from the current `catalog.settings.json`** (those are curated, not generated). Overwrite `summary` and add a new `liveFields` array from the schema's `properties`. New schemas (not in the current catalog) get a default `family: misc` that a human must reassign during review.
4. Write the new `catalog.settings.json`. The byte-exact `-Check` from Design 001 / current PR #8 applies.
5. Run `Sync-ConfigCatalog.ps1` to regenerate `modules/configs/`. This step is byte-deterministic across PS editions (PR #8 fix).
6. Exit non-zero if anything in step 2 fails (so a Dynatrace platform incident doesn't silently produce a stale catalog).

### Scheduled GitHub Action — `.github/workflows/catalog-refresh.yml`

```yaml
name: catalog-refresh
on:
  schedule:
    - cron: '17 6 * * MON'   # Mondays 06:17 UTC
  workflow_dispatch:

jobs:
  refresh:
    runs-on: ubuntu-latest
    environment: catalog-refresh   # gated environment; carries DT_PLATFORM_TOKEN
    permissions:
      contents: write              # to push a branch
      pull-requests: write         # to open the PR
    steps:
      - checkout
      - install pinned Monaco (reuse the SHA-256-verified install pattern)
      - run Sync-CatalogFromSchemas.ps1 with DT_ENVIRONMENT + DT_PLATFORM_TOKEN from the gated environment
      - if anything changed: create branch chore/catalog-refresh-<YYYY-MM-DD>, commit, open PR titled
        "chore(catalog): refresh reflected schemas - <date>" with @copilot review requested
```

The cron uses Dynatrace credentials with read-only schema scope only (`settings:schemas:read`). No write scopes; no environment URLs that could be mutated.

### Review template — auto-PR body

The cron opens the PR with a templated body:

```markdown
## Auto-generated catalog refresh

| | |
|---|---|
| Trigger | scheduled (cron) / workflow_dispatch |
| Source environment | <DT_ENVIRONMENT> |
| Monaco version | <version> |
| Schemas in scope | <N> |

### Diff summary

- Updated summaries: <N> entries
- New live fields: <N> entries
- New schemas (need family assignment): <list of IDs with family: misc>
- Schemas no longer resolvable (deprecated upstream?): <list>

### Reviewer checklist

- [ ] Diffstat looks mechanical (no surprise structural changes)
- [ ] New schemas have been assigned the right `family` (not `misc`)
- [ ] Any "no longer resolvable" entries are either removed from schemas.txt or have the deprecation noted in CHANGELOG
- [ ] CI is green (the byte-exact catalog `-Check` validates the regen)
```

A human merges. If the diff is large or surprising, a human can also push fixes onto the auto-PR branch before merging — the same PR loop, with Copilot review.

## Migration / rollout

| Step | What |
|---|---|
| 1 | This proposal merges |
| 2 | Implementation PR lands: `Sync-CatalogFromSchemas.ps1`, `schemas.txt`, the workflow. Crucially the FIRST run of the workflow is via `workflow_dispatch`, with the gated environment unconfigured — so the run exits early with "no creds; nothing to do". This verifies the workflow shape without touching Dynatrace. |
| 3 | A maintainer provisions a read-only Dynatrace platform token (`settings:schemas:read` only), creates the `catalog-refresh` GitHub Actions environment, and stores the secrets. Reviewer count on the environment ≥ 1. |
| 4 | A maintainer manually triggers `workflow_dispatch` once. Either no diff (catalog already matches live) or a small refresh PR opens. Merge it as the first refresh. |
| 5 | The Monday cron takes over. |

## Alternatives considered

- **Auto-merge the refresh PR if CI passes.** Rejected: the value of the auto-PR is the human review, especially for new schemas needing family assignment. Auto-merge undermines the gate that's the whole point.
- **A monolithic `monaco generate schema` dump committed to the repo, no curated catalog.** Rejected: that's the "ossify by including everything" anti-pattern. The schemas.txt curation is what keeps the catalog opinionated.
- **A pre-commit hook that refreshes the catalog before every push.** Rejected: requires every developer to have Monaco + valid Dynatrace creds locally, which violates the harness's "wrappers work offline" principle.
- **GitHub App that runs the refresh on-demand from a chat command.** Rejected: extra infra to maintain (the App), extra auth surface (App installation tokens), no clear win over a scheduled workflow plus `workflow_dispatch`.

## Open questions

1. **Cron frequency.** Weekly seems right (Dynatrace schemas don't change daily; PR fatigue if more frequent). Confirm with the consumer.
2. **What does the cron do if the source Dynatrace environment is unreachable?** Proposal: exit 0 (no diff, no PR), log loudly, and rely on the operator to notice. Failing the workflow run would page someone weekly during any DT incident, which is wrong.
3. **Should the auto-PR include a regenerated `docs/CONFIG-COVERAGE.md` "What's in the catalog today" table?** Proposal: yes. The table is derived from the catalog; keeping it in sync via the cron is free.
4. **Do we need separate inputs files per environment** (e.g., a dev tenant has experimental schemas that prod doesn't)? Proposal: no. One curated inputs file; if you need experimental schemas, branch the inputs file in a feature PR.

## Acceptance criteria

The implementation PR is mergeable when **all** of the following hold:

- [ ] `config/catalog/schemas.txt` exists with the current 8 Monaco catalog IDs.
- [ ] `scripts/monaco/Sync-CatalogFromSchemas.ps1` exists, with comment-based help and `-WhatIf` support so it can be exercised in CI without writing files.
- [ ] `.github/workflows/catalog-refresh.yml` exists, runs only on `schedule` and `workflow_dispatch`, scoped to a `catalog-refresh` GitHub Actions environment.
- [ ] The workflow opens a PR via `gh pr create` with the standard PR template body plus the auto-generated diff summary.
- [ ] The workflow requests `@copilot` review on the auto-PR.
- [ ] A Pester test in `tests/Harness.Tests.ps1` exercises `Sync-CatalogFromSchemas.ps1 -WhatIf` against a stubbed `monaco generate schema` output and asserts the resulting catalog is byte-identical to a fixture.
- [ ] CHANGELOG updated.
- [ ] First `workflow_dispatch` run is documented as a manual operator step (it requires the maintainer to provision the GitHub Actions environment + secrets).
